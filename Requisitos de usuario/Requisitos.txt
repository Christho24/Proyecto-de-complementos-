Caso: Plataforma Finis Trabaja

Actores:
     - Estudiante
     - Empresa externa
     - Área interna de la universidad
     - Administrador
     - Visitante

Asunciones:
     - La plataforma será utilizada por estudiantes de la Universidad Finis Terrae.
     - Las empresas externas y áreas internas de la universidad podrán registrarse para publicar ofertas.
     - El administrador será responsable de validar a las empresas antes de que publiquen vacantes.
     - Las ofertas publicadas podrán ser vistas por cualquier persona que acceda a la plataforma.
     - Los estudiantes deberán tener un perfil CVV para postular desde la plataforma.
     - El sistema contará con un módulo recomendador para relacionar ofertas con perfiles de estudiantes.
     - La plataforma será una aplicación web simple y funcional, ejecutada de forma local para la evaluación del proyecto.

Requisitos Funcionales (RF --> F):

---

RU01: El usuario se registra en la plataforma. (RF)
     RS01: Registrar Usuario indicando el tipo: Estudiante, Empresa, Área interna.(RF)
     RS02: Registrar correo y contraseña.(RF)
     RS03: Validar correo (RF)
     RS04: Guardar los datos de registro.(RF)
     RS05: Dejar pendiente de validación a empresas externas y áreas internas.(RF)

---

RU02: El usuario inicia sesión en la plataforma. (RF)
     RS06: Solicitar correo y contraseña. (RF)
     RS07: Validar credenciales ingresadas.(RF)
     RS08: Identificar tipo de usuario: estudiante, empresa, área interna o administrador.(RF)
     RS09: Permitir acceso según permisos del usuario.(RF)
     RS10: Denegar acceso si los datos son incorrectos.(RF)

---

RU03: El estudiante crea su Currículum Vitae Virtual CVV.(RF)
     RS13: Registrar datos personales básicos. (RF)
     RS14: Registrar carrera que estudia.(RF)
     RS15: Registrar especialidades y áreas de conocimiento.(RF)
     RS16: Registrar roles o tipos de cargo de interés.(RF)
     RS17: Registrar pretensión de sueldo mínima aceptable.(RF)
     RS18: Registrar disponibilidad de jornada.(RF)
     RS19: Registrar preferencia de modalidad: presencial, remota o híbrida.(RF)
     RS20: Registrar experiencias laborales previas. (RF)
     RS21: Registrar logros académicos o extracurriculares. (RF)
     RS22: Adjuntar CV en PDF, portafolio o certificados. (RF)
     RS23: Configurar visibilidad del perfil.(RF)

---

RU04: El administrador valida empresas o áreas internas registradas. (RF)
     RS24: Revisar solicitudes de registro. (RF)
     RS25: Aprobar empresa externa o área interna. (RF)
     RS26: Rechazar empresa externa o área interna. (RF)
     RS27: Habilitar publicación de ofertas cuando el registro sea aprobado. (RF)

---

RU05: La empresa o área interna publica una oferta laboral.(RF)
     RS28: Indicar rol o título del puesto. (RF)
     RS29: Describir funciones y responsabilidades. (RF)
     RS30: Especificar carrera o carreras a las que va dirigida. (RF)
     RS31: Señalar rango de sueldo ofrecido. (RF)
     RS32: Definir jornada: tiempo completo o parcial. (RF)
     RS33: Indicar ubicación del trabajo. (RF)
     RS34: Indicar modalidad: presencial, remota o híbrida. (RF)
     RS35: Adjuntar archivos o imágenes a la oferta. (RF)

---

RU06: El visitante visualiza ofertas laborales.(RF)
     RS36: Mostrar ofertas publicadas sin necesidad de registro. (RF)
     RS37: Filtrar ofertas por carrera. (RF)
     RS38: Filtrar ofertas por rango de sueldo. (RF)
     RS39: Filtrar ofertas por tipo de rol. (RF)
     RS40: Filtrar ofertas por ubicación geográfica o modalidad. (RF)
     RS41: Filtrar ofertas por tipo de jornada. (RF)

---

RU07: El sistema recomienda estudiantes para una oferta. (RF)
     RS42: Cruzar información de la oferta con perfiles CVV.  (RF)
     RS43: Comparar carrera requerida con carrera del estudiante.  (RF)
     RS44: Comparar rol ofrecido con roles de interés del estudiante. (RF)
     RS45: Comparar sueldo ofrecido con pretensión del estudiante. (RF)
     RS46: Comparar jornada y modalidad con disponibilidad del estudiante. (RF)
     RS47: Calcular grado de compatibilidad. (RF)
     RS48: Generar listado ordenado de estudiantes recomendados. (RF)

---

RU08: El administrador revisa recomendaciones.(RF)
     RS49: Acceder al listado de estudiantes recomendados por oferta. (RF)
     RS50: Visualizar grado de compatibilidad de cada estudiante. (RF)
     RS51: Compartir recomendaciones con la empresa oferente si la universidad lo decide. (RF)

---

RU09: El estudiante postula a una oferta laboral. (RF)
     RS52: Seleccionar una oferta de interés. (RF)
     RS53: Postular directamente desde la plataforma. (RF)
     RS54: Adjuntar opcionalmente carta de presentación. (RF)
     RS55: Registrar postulación del estudiante. (RF)

---

RU10: La empresa revisa postulantes. (RF)
     RS56: Acceder al listado de postulantes de cada oferta. (RF)
     RS57: Visualizar el CVV de cada postulante. (RF)

Requisitos No Funcionales (RNF )

RNF de Producto (calidad de software)

     RNF-Usability01: La plataforma debe tener una interfaz clara y simple para estudiantes, empresas y administrador.
     RNF-Usability02: La creación del CVV debe ser comprensible para el estudiante.
     RNF-Usability03: La publicación de ofertas debe ser fácil de completar para empresas y áreas internas.
     RNF-Usability04: Los filtros de búsqueda deben ser visibles y fáciles de aplicar.
     RNF-Usability05: Los mensajes del sistema deben ser claros para el usuario.

     RNF-Reliability01: El sistema debe mantener correctamente registrados los perfiles CVV.
     RNF-Reliability02: El sistema debe mantener correctamente registradas las ofertas laborales.
     RNF-Reliability03: El sistema debe evitar la pérdida de postulaciones realizadas.
     RNF-Reliability04: El sistema debe mantener consistencia entre ofertas, postulantes y empresas.
     RNF-Reliability05: El sistema debe permitir recuperar información ante fallos simples.

     RNF-Performance01: El listado de ofertas debe cargar en un tiempo razonable.
     RNF-Performance02: Los filtros deben responder sin demoras excesivas.
     RNF-Performance03: El módulo recomendador debe generar resultados en un tiempo adecuado.
     RNF-Performance04: La plataforma debe permitir el uso simultáneo de varios usuarios sin afectar gravemente su funcionamiento.

     RNF-Supportability01: El código fuente debe estar ordenado y ser comprensible.
     RNF-Supportability02: Los archivos del proyecto deben tener nombres coherentes.
     RNF-Supportability03: La plataforma debe poder ejecutarse de forma local.
     RNF-Supportability04: El sistema debe permitir futuras modificaciones o mejoras.
     RNF-Supportability05: El proyecto debe incluir los archivos necesarios para su ejecución.

RNF Organizacionales y Externos

     RNF-Other01: La plataforma debe proteger los datos personales de estudiantes y empresas.
     RNF-Other02: El acceso a la información debe depender del tipo de usuario.
     RNF-Other03: Las empresas solo podrán publicar ofertas después de ser validadas por el administrador.
     RNF-Other04: El sistema debe funcionar como aplicación web navegable.
     RNF-Other05: La plataforma debe ser funcional para la evaluación local del proyecto.
     RNF-Other06: El sistema debe permitir trazabilidad básica de perfiles, ofertas y postulaciones.
